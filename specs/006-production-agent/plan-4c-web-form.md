## Summary

Phase 4C-iii adds the third and final intake channel: a production-quality Next.js 15 web
support form at `src/web-form/`. It delivers 4 pages (landing `/`, support form `/support`,
ticket status `/ticket/[id]`, dashboard `/dashboard`), 3 new FastAPI endpoints, 3 Next.js
API proxy routes, and 5 shared UI components. The form is the **highest scoring item
(10 points)** in the hackathon rubric.

**Dependency order (hard constraint)**:
Block 1 (FastAPI endpoints) → Block 4 (Next.js proxies) can only be fully validated after
Block 1. Blocks 2–3 (scaffold + components) can proceed in parallel with Block 1.

## Technical Context

| Item | Decision |
|------|----------|
| Language/Version | Python 3.12 (FastAPI) + TypeScript 5 (Next.js) |

### B1-T2 — New DB Queries: Ticket Lookup + Metrics
**File**: `production/database/queries.py` (update)
**Risk**: LOW — additive query functions, no changes to existing functions

- [ ] B1-T2.1 Add `get_ticket_by_display_id(pool, ticket_id: str) -> dict | None`:
  - If `ticket_id.startswith("TKT-")`: query `WHERE upper(substring(id::text, 1, 8)) = $1`
    (extract 8-char suffix after "TKT-")
  - Else: query `WHERE id::text = $1` (UUID lookup)
  - JOIN `customers` on `customer_id` to get `name`, `email`
  - JOIN `messages` (first `role='customer'` message) to get original message body
  - Return dict with: `ticket_id` (display), `internal_id`, `status`, `category`, `priority`,
    `subject`, `message`, `customer_name`, `customer_email`, `created_at`, `updated_at`,
    `resolved_at`
  - Compute display ID: `'TKT-' + str(row['id'])[:8].upper()`
- [ ] B1-T2.2 Add `get_metrics_summary(pool) -> dict`:
  - Total/open/in_progress/resolved/escalated counts (single COUNT FILTER query)
  - Channel breakdown (GROUP BY channel)
  - Recent 10 tickets (JOIN customers, ORDER BY created_at DESC, LIMIT 10)
  - Compute `escalation_rate = round((escalated / total) * 100, 1)` if total > 0 else 0.0
  - Return complete dict per `contracts/web-form-api.md`

### B1-T3 — WebFormHandler ⚠️ HIGH RISK
**File**: `production/channels/web_form_handler.py` (replace stub)
**Risk**: HIGH — multi-step pipeline: customer lookup → conversation → message → ticket → Kafka.
Any failure in the middle leaves orphaned records. Rollback strategy: all steps in one try/except.

- [ ] B1-T3.1 Define `WebFormInput` Pydantic v2 model:
  ```python
  class WebFormInput(BaseModel):
      name: str = Field(min_length=1, max_length=200)
      email: EmailStr
      subject: str = Field(min_length=5, max_length=200)
      category: Literal["billing", "technical", "account", "general"]
      priority: Literal["low", "medium", "high", "urgent"]
      message: str = Field(min_length=20, max_length=2000)
  ```
- [ ] B1-T3.2 Implement `submit_ticket(pool, body: WebFormInput) -> dict`:
  1. `get_or_create_customer(pool, email=body.email, name=body.name)` → `customer`
  2. `create_conversation(pool, customer_id=customer['id'], channel="web_form")` → `conv_id`
  3. `add_message(pool, conv_id, role="customer", content=body.message, channel="web_form")` → `msg_id`
  4. `create_ticket(pool, conv_id, customer['id'], channel="web_form", subject=body.subject, category=body.category, priority=body.priority)` → `ticket_id_uuid`
  5. Build `TicketMessage(id=str(uuid4()), channel="web_form", customer_email=body.email, customer_name=body.name, subject=body.subject, message=body.message, received_at=datetime.now(ZoneInfo("Asia/Karachi")).isoformat(), metadata={})` 
  6. `await kafka_producer.publish_ticket(ticket_message)` — on Kafka failure: log to stderr, do NOT raise (ticket already created in DB)
  7. Return `{ "ticket_id": "TKT-" + ticket_id_uuid[:8].upper(), "internal_id": ticket_id_uuid, "status": "open", "created_at": ... }`
- [ ] B1-T3.3 Wrap entire `submit_ticket` in `try/except`; on exception return `None` and log

## Risks & Follow-ups

1. **FastAPI server must be running for end-to-end testing.** During Next.js development,
   `FASTAPI_URL` must point to a running FastAPI process (`uvicorn production.api.main:app
   --port 8000`). Use mock API responses (via `jest` or `msw`) if FastAPI isn't available.

2. **Kafka publish failure is non-fatal for web form.** The ticket is persisted in PostgreSQL
   before Kafka publish is attempted. If Kafka is unavailable, the customer still receives their
   ticket ID, but the AI agent won't process it until Kafka recovers. This is documented behaviour
   for hackathon phase.

3. **Priority field migration.** B1-T1 must be applied to the Neon PostgreSQL instance before
   B1-T3 can function. Verify the migration was applied before running integration tests.
